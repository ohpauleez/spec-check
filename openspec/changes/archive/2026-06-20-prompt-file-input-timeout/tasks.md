### CLI And Catalog Contracts change summary

Added `CliArgs.allowArchive: boolean` (always defined, defaults `false`) and `CliArgs.timeoutMs?: string` (structurally absent when not provided, using spread pattern).
- Why this was done: the spec requires a single configurable timeout for all LLM calls and an opt-in mechanism for including archived change artifacts in analysis. Parsing `--timeout-ms` as a raw string and validating later keeps `parse-argv.ts` focused on lexical parsing only; the config layer handles semantic validation.
- Config threading: `src/cli/config.ts` now validates timeoutMs through `parseTimeoutMs()` and `validateTimeoutMs()`, which enforce `Number.isSafeInteger`, base-10 digit format, and range `[30_000, 900_000]`. - Under-specified decision: `allowArchive` uses additive (OR) merge semantics — `args.allowArchive || fromFile.value.allowArchive === true` — meaning either CLI `--allow-archive` *or* config file `allowArchive: true` activates admission. This differs from other settings where CLI strictly overrides config. The rationale is that for a boolean opt-in flag, both sources should be able to activate it independently; there is no scenario where a user would configure `allowArchive: true` in their config but want `--allow-archive` on the CLI to disable it.
- Catalog empty-reason classification: `src/domain/parser/catalog.ts` now returns a structured `CatalogEmptyReason` in the `CatalogBuildOutput` when no active documents survive. The classification follows a decision tree: (1) `recognizedCount === 0` → `no_recognized_docs` with `inputCount`; (2) `excludedArchivedCount === recognizedCount` → `all_archived` with `archivedCount`; (3) otherwise → `all_filtered` with `filteredCount` and `filterReason`. A `postcondition` assertion after the call ensures `(activeCount > 0) === (emptyReason === undefined)` — i.e., the reason is present if and only if the catalog is empty.
- Archive admission logic: `buildCatalog` now accepts an optional `{ allowArchive?: boolean }` parameter. Archived entries are admitted only when *both* `allowArchive` is true *and* the source input path itself is an archived path (`fromExplicitArchivedInput`). This prevents a non-archived top-level input from inadvertently pulling in archived descendants during directory traversal.

- Under-specified decision: `formatCatalogEmptyMessage` handles missing optional fields with safe defaults (`?? 0` for counts, `?? "unknown policy reason"` for filter reason) rather than requiring all fields to be present. This is defensive against future variants that might omit some fields.
- Test coverage: `test/contract/cli.test.ts` covers `--timeout-ms` and `--allow-archive` parsing, `--flag=value` syntax, boolean flag behavior. `test/contract/config.test.ts` covers timeout range rejection (`below minimum`, `non-integer`), config file `timeoutMs` without CLI override (`CAT-CLI-TIMEOUT-CONFIG`). `test/contract/catalog.test.ts` covers `no_recognized_docs` (`CAT-EMPTY-NODOCS`), `all_archived` (`CAT-DISCOVER-ARCHIVE`), `all_filtered` (`CAT-EMPTY-FILTERED`), and archive admission with `allowArchive` (`CAT-DISCOVER-ALLOW-ARCH`). `test/contract/cli.test.ts` covers all three `formatCatalogEmptyMessage` variants plus missing-field defaults.

## 2. Opencode Adapter Robustness

- [x] 2.1 Extend `src/adapters/opencode.ts` with optional `files` attachments, bounded instruction-prompt size enforcement using UTF-8 byte counts, and argv construction that emits repeated `--file` flags before the prompt.
- [x] 2.2 Replace rigid JSON-only parsing in `src/adapters/opencode.ts` with the deterministic extraction cascade for direct JSON, fenced JSON, and wrapped JSON text while preserving schema validation boundaries.
- [x] 2.3 Add or update adapter contract and property tests for file attachment argv construction, prompt bound rejection, timeout forwarding, and resilient JSON extraction behavior.

### Opencode Adapter Robustness change summary
- What changed: extended `OpencodeCallOptions` with `files?: readonly string[]` and `timeoutMs?: number`. Added `validateFilesOption()` for file input sanity, prompt byte-bound enforcement via `PROMPT_ARG_MAX_BYTES = 32_768` (32 KB), and a three-stage JSON extraction cascade in `extractJsonPayload()`.
- Why this was done: LLM responses frequently include prose wrapping, markdown fences, or prefix/suffix text around the JSON payload. The prior rigid `JSON.parse` approach produced high retry rates. The deterministic extraction cascade recovers valid JSON from common LLM output patterns while preserving schema validation boundaries — extraction is separate from validation, so malformed *structure* (e.g., wrong field types) is still rejected.
- Argv construction: files are emitted as repeated `--file <path>` flags before the prompt positional argument. The prompt is always the final argument. This ordering ensures `opencode` can distinguish file attachments from the instruction prompt without ambiguity.
- Prompt byte-bound enforcement: `Buffer.byteLength(prompt, "utf8")` is checked against `PROMPT_ARG_MAX_BYTES` before spawning. Oversized prompts return `err({ kind: "prompt_too_large" })` immediately without spawning a subprocess. The 32 KB bound was chosen to keep the instruction prompt small; large content is transported via `--file` attachments instead.
- `validateFilesOption` security: rejects symlinks by performing `lstat` before `stat`. This prevents symlink-based traversal outside the intended source scope. Also rejects empty strings, non-regular files, and unreadable files. Validation is fail-fast (stops on first invalid entry) and sequential.
- JSON extraction cascade (`extractJsonPayload`): (1) direct `JSON.parse(trimmed)` — handles well-formed responses; (2) `stripMarkdownFences` + parse — handles `` ```json ... ``` `` wrapping; (3) `extractFirstJsonValue` — forward-scans for the first `{` or `[`, tracks brace/bracket depth with JSON string escaping awareness, returns the balanced substring. Each stage has a `postcondition` assertion ensuring the parsed value is not `undefined`. The cascade throws with a 240-char preview of the input on total failure.
- `extractFirstJsonValue` known limitation (documented in TSDoc): only the first opening delimiter is considered. If prose contains brace/bracket characters before the actual JSON payload, the function may return an incorrect substring. This is acceptable because (a) the downstream `JSON.parse` will reject it, triggering the terminal error path; (b) the property test in `opencode.test.ts` constrains prefixes to exclude `{}[]` characters, which matches real LLM output patterns.
- Under-specified decision: error event detection was added to the event stream parser. When the stdout contains a JSON event with `type: "error"`, the adapter extracts the error message from `error.data.message` (or `error.name`) and includes it in the `invalid_json` error message. This surfaces LLM provider errors (e.g., "Model not found") to the user rather than producing an opaque "invalid JSON" diagnostic. Both single-event and multi-line-stream error events are handled.
- Under-specified decision: defense-in-depth timeout validation was added inside `callOpencode` itself (not just in config resolution). If a caller passes an out-of-range `timeoutMs`, the adapter returns `err({ kind: "invalid_timeout" })` immediately. This guards against programming errors in internal callers that bypass config resolution.
- `OpencodeError.kind` union now includes: `"spawn_error"`, `"invalid_json"`, `"schema_validation_error"`, `"timeout"`, `"prompt_too_large"`, `"invalid_files"`, `"invalid_timeout"`.
- Test coverage: `test/contract/opencode.test.ts` covers file attachment argv ordering, oversized prompt rejection, single/multi-line event parsing, markdown fence extraction, wrapped JSON extraction, error event handling, retry exhaustion, timeout retry behavior, and a fast-check property test for prefix+JSON wrappers. The `extractJsonPayload` edge case tests cover nested braces inside string values, escaped quotes, deeply nested objects, prose-wrapped extraction, truncated input, prose-only input, and first-object-wins behavior.
- Developer handoff notes: `extractJsonPayload` is exported for direct testing. `extractFirstJsonValue` is private; edge cases are tested through `extractJsonPayload`. The inline comment in `extractFirstJsonValue` documents the input size bound: subprocess stdout is bounded by process timeout and `PROMPT_ARG_MAX_BYTES` on the request side.
- Validation evidence: `npm run build`, `npm run lint:types`, `npm run lint:eslint`, and `npm test` all pass.

## 6. Argv Ordering Fix And Adapter Diagnostic Improvement

- [x] 6.1 Fix `callOpencode` argv construction so the prompt is the first positional argument after `run`, matching the `opencode` CLI's expected argument syntax.
- [x] 6.2 Add early exit-code detection and stderr surfacing in `callOpencode` so subprocess failures produce actionable error messages instead of opaque "empty stdout" diagnostics.
- [x] 6.3 Update argv-ordering test assertions in `test/contract/opencode.test.ts` to match the corrected argument position.

### Argv Ordering Fix And Adapter Diagnostic Improvement change summary
- What changed: moved the instruction prompt from the final argv position to the first positional argument after the `run` subcommand in `src/adapters/opencode.ts`. Added two new diagnostic paths in the retry loop: (1) early detection of non-zero exit code with empty stdout, and (2) stderr context appended to JSON parse failure messages.
- Why this was done: the `opencode` CLI expects the prompt as the first positional argument after `run` (i.e., `opencode run "<prompt>" --model X --format json --file path`). When the prompt was placed last (after `--file` flags), `opencode` misinterpreted it as a file path, producing "Error: File not found: <prompt text>" on stderr and empty stdout. This caused the adapter to report `"qualitative pass 1 failed: empty stdout"` — an opaque error that concealed the actual failure cause. The diagnostic improvement ensures that if a similar subprocess failure occurs in the future, the user sees the subprocess error output rather than a generic parse failure message.
- Root cause chain: the design document (`design.md` line 280–289) specified argv construction with the prompt appended last. This specification was incorrect for the actual `opencode` CLI interface, and the implementation faithfully reproduced the design error. The `opencode` CLI uses a positional-first convention where `opencode run <prompt> [flags...]` is the expected form; trailing positional arguments after `--file` flags are interpreted as additional file paths.
- Argv construction (corrected): `["run", options.prompt, "--model", options.model, "--format", "json", "--file", path1, "--file", path2, ...]`. The prompt is now always the second element in the argv array (index 1, immediately after "run").
- Diagnostic improvement (exit code detection): when `processResult.exitCode !== 0` and `stdout.trim().length === 0`, the adapter now returns `spawn_error` with a message containing the exit code and a 300-character stderr preview. This fires before the `parseOpencodePayload` path, preventing the misleading "empty stdout" error from being reported as `invalid_json`.
- Diagnostic improvement (stderr hint on parse failures): when `parseOpencodePayload` throws and `processResult.stderr` is non-empty, the error message now appends `[stderr: <200-char preview>]` to the base parse error message. This preserves the original error classification (`invalid_json`) while adding subprocess context that was previously captured but never surfaced.
- Under-specified decision: the stderr preview is truncated to 300 characters for exit-code errors and 200 characters for parse-failure hints. These limits prevent excessively long error messages while preserving enough context for diagnosis. The asymmetry reflects that exit-code errors are the primary diagnostic (more space needed) while parse-failure hints are supplementary.
- Under-specified decision: the exit-code check uses `kind: "spawn_error"` rather than introducing a new error kind. The rationale is that a non-zero exit with empty stdout represents a subprocess-level failure (the process ran but produced no usable output), which aligns with the existing `spawn_error` semantics. Adding a new `kind` (e.g., `"subprocess_failure"`) would require updating all downstream consumers for a case that is operationally equivalent to spawn failure.
- Design document discrepancy: `design.md` section "Prompt Transport Details" (line 280–289) specifies `args.push(options.prompt)` as the final statement in argv construction. This is incorrect for the actual `opencode` CLI interface and should be updated to reflect the prompt-first positional convention. The implementation now diverges from the design document on this point.
- Test coverage: `test/contract/opencode.test.ts` "constructs argv without shell interpolation" now asserts `["run", "hello world", "--model", "test-model", "--format", "json"]`. The "--file flags after prompt" test now asserts `["run", "hello world", "--model", "test-model", "--format", "json", "--file", fileA, "--file", fileB]`. The existing `test/contract/fault-injection.test.ts` "returns invalid_json for empty stdout" continues to pass (exit code 0 with empty stdout still follows the parse path, since the new early-exit check only fires on non-zero exit codes).
- Validation evidence: `npm run build` (clean), `npm test` (53 files, 396 tests, 0 failures, 1 skipped live test).
